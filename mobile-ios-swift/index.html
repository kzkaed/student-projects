Navigating Xcode

Navigator - Left hand Side

The navigator shows all the project files in the app. With the top bar of the navigator you can search, look at the hierarchy of the app and its funtions, or look at old builds.

Utilities - Right hand Side

The Utilities area contains, in the top part, an inspector where you can analyse and change attributes. The bottom section is a palette of UI components. This pallete's components will be dragged to the Main.storyborad where you will layout your design work.

Main.storyboard

This is where we will set up our layout or design. We will use components of the UI pallette and adjust their attributes in the inspector.

ViewController.swift

The ViewController class is where all the glue goes to make the view (UI) actionable. The controller will hold the connection from the storyboard and handle what happens. Each View will have at least on Controller. You will notice at the top of the controller UIKit. This is the library (a group of code) that has all the UI code. The controller is where we are going to drap our UI components to wire them up.

Process to connect UI View to the Controller

Click on UI component, press control button on keyboard, and drag to controller.

We will goes through this process in detail. If you were a designer and you only wanted to work on the layout of the application, bringing the connections over to the view for someone to program would be a great first step. Of course, at that point, you could also code what happens. We will do some of that coding today as well.

Assets.xcassets

Xcode assests. This is where we will import the images for the project

MVC

MVC stands for Model View Controller. It is a software design pattern. It is how xcode was designed and it is enforced in the way you will build your application.

Model is what the program does that is independent of the UI. It is the data.

View is the user interface, the buttons, display, images, videos, etc that you see on the screen of the device.

Controller acts as the glue between the view and model.

The view and the model never talk to each other. The controller is the 'middleperson'. The controller will ask the model for its data. The controller or one of its sub classes will handle that data to pass to the view.
